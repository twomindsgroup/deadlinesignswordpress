### Finding 6 — Root cause of the 43 GB spam (identified, not yet fixed)

In `wc-special-product-pricing`, two `session_start()` call sites at `inc/functions.php:7` and `inc/ajax.php:10`:

- Both guard with `session_status() === PHP_SESSION_NONE`
- Neither checks `headers_sent()`

When any output occurs before `session_start()` (Wordfence WAF auto-prepend, PHP 7.4 deprecation notices, other plugin output), session_start fires the "Cannot send session cache limiter" warning every time.

With 100+ active plugins on WP 5.0.4 / PHP 7.4, this fires on every anonymous pageload, 24/7 — hence 210M lines.

**Fix options identified:**

- **Option A (minimum):** Add `!headers_sent() &&` to both guards + `isset($_SESSION['nonlogin_form_data'])` check at `ajax.php:13`. Surgical, preserves `$_SESSION` usage. Leaves latent bug where if headers ARE sent, `$_SESSION` never initializes and writes fail silently.
- **Option B (proper):** Migrate `$_SESSION` → `WC()->session` for the 2 keys (`_token_auto_delete`, `nonlogin_form_data`). Plugin already uses `WC()->session` in `inc/add-fee.php:18-25`, so migration aligns existing patterns. 7 call sites across 2 files. Eliminates `session_start()` entirely.
- **Recommendation:** Option B. Deferred because prod is PHP 7.4/WC 3.6.2 but current staging is PHP 8.1/WC 10.5.2 — version parity too different for safe validation.
